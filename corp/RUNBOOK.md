# Syntropia Research RUNBOOK

書籍会社のPhase制とは異なり、本会社は継続運転モデルである。ルーチンは日次/週次/イベント駆動の
3種類で構成する。各ルーチンは対応する `playbooks/*.md` のチェックリストに従って実行する。
本運用ルールは実運転しながら `corp/board.md` の決定ログで固めていく（本ファイルの細部は
随時更新される想定であり、変更は自動push可）。

## 前提

- 全てのルーチンは絶対制約5条（`corp/CLAUDE.md` 第3章）に従う。
- 全ての成果物には front-matter に `as_of`（分析基準日時）を必須記載する。
- リスク上限値の正本は `corp/board.md`（D-002, D-003）。playbook側の数値と食い違う場合は
  board.mdを正とする。
- KPI集計・ダッシュボード生成はCEOが要求した時のみ実施する（自発生成禁止）。

## 日次ルーチン

トリガー: 毎営業日。

1. **スクリーニング**（`playbooks/screening.md`）: 定量フィルタによる一次候補の抽出。
2. **イベント検知**（D-006）: 保有銘柄・WATCH銘柄を対象にEDINET・適時開示情報を巡回する。
   決算発表・適時開示・株式分割・増資等の重大イベントを検知した場合、当日中にイベント駆動
   ルーチンへ引き継ぐ。
3. 保有中ポジション・WATCHリストの株価鮮度チェック（数値実在原則に基づき、取得元・取得日時を
   更新）。
4. 前日までに記録部（`playbooks/logging.md`）へ記録された約定・株価変動から、
   リスク上限（D-002）への抵触がないかの機械照合（`playbooks/risk-check.md`）。

## 週次ルーチン

トリガー: 週1回、**土日**に実施する（D-005）。

1. 日次スクリーニングで積み上がった候補のうち、深掘り対象を企業調査部（`playbooks/research.md`）
   へ送る。
2. 企業調査部の成果物を受けて投資判断部（`playbooks/judgement.md`）が判断メモ（BUY/WATCH/PASS）
   を起案・校閲（1稿＋1校閲）。
3. リスク管理部（`playbooks/risk-check.md`）が全判断メモをD-002/D-003の上限値と機械照合。
   逸脱案はこの時点で提案自体を却下する。
4. 却下されなかった判断メモをCEOへ提示（発注はCEOが手動で行う）。
5. 記録部（`playbooks/logging.md`）が週間サマリ（提案数・機械照合の却下件数など）を記録する。
   ※これはKPIダッシュボードではなく、記録部の定型ログである。

## イベント駆動ルーチン

トリガー: 日次ルーチンのイベント検知（D-006）で決算発表・適時開示・株式分割・増資等の
重大イベントが見つかった場合。

1. 保有銘柄・WATCHリストの対象銘柄でイベントを検知した場合、当日中に企業調査部
   （`playbooks/research.md`）へ緊急調査依頼を発行する。
2. 投資判断部が既存の判断メモを更新するか、新規に起案するかを判断する
   （`playbooks/judgement.md`）。更新の場合も「1稿＋1校閲」を維持する。
3. リスク管理部が更新後の内容を再度機械照合する。
4. CEOへ速報として提示する。

## 記録部のフィードバックループ（唯一の例外）

- 記録部は実際の売買と結果（勝敗・想定シナリオとの乖離）をログする。
- 記録部が「次のスクリーニング条件を変えるべき」と判断した場合でも、エージェントは
  スクリーニング条件を直接書き換えない。
- 教訓案をCEOに提示し、CEO承認後に `corp/board.md` へ新しい決定ID（D-XXX）として記録された
  ものだけを `playbooks/screening.md` に反映する。

## 既知の技術的制約（診断確定・引き継ぎ事項）

初回スクリーニングの試験実行（Phase0完了直後）で発覚した、D-003（Web取得全面許可）の
実行可能性に関する技術的ブロッカー。**2026-07-09に原因が確定した。ネットワークポリシーの
問題ではなく、`WebFetch` ツール固有のリクエスト（User-Agent等）を一部サイトがボットとして
弾いている、サイト側のbot対策が原因だった。**

### 確定した診断（2026-07-09、別セッションでの検証＋本セッションでのcurl再現）

- `jpx.co.jp`・`kabutan.jp` は `WebFetch` では403だが、**`curl`（プロキシ経由・環境変数変更なし）
  では200 OK**。プロキシの `noProxy` リストにも変化はなく、遮断リストにも入っていない
  （`__agentproxy/status` の `recentRelayFailures` は空）。
- つまり **到達性そのものは最初から問題なかった**。CEOに依頼した環境設定変更（ネットワーク
  ポリシー）は今回の403とは無関係だった可能性が高い。Yahoo!ファイナンス等が新セッションで
  復旧したのは、環境変更のタイミングと偶然重なったか、あるいはサイト側のbot判定が一時的
  だったなど別要因の可能性がある（未検証・結論づけない）。
- **ワークアラウンド**: `jpx.co.jp` / `kabutan.jp` 等、`WebFetch` が403を返すサイトは、
  `Bash` で `curl`（必要ならブラウザ相当の `-A` User-Agent指定）を使い、取得したHTMLを
  読んで内容を確認する方式に切り替えられる。ただし `curl` はHTMLを生で返すため、
  `WebFetch` のようなAI要約は行われない点に注意（自分でパース・要約する）。

### Yahoo!ファイナンスの一時的な500エラー（2026-07-09、企業調査部RSH-S06で発生）

5803（フジクラ）の企業調査中、`finance.yahoo.co.jp` がWebFetch・curlの両方で終日500エラーを
返し取得不能だった事象を確認。既知の403問題（bot対策）とは別種のエラーで、Yahoo側の一時的な
不調の可能性が高い。代替として株探（kabutan.jp、curl経由）で必要データを取得できたため
作業は継続できたが、**Yahoo!ファイナンスが唯一のデータソースになっている工程では500エラー時の
代替手段（kabutan等への切り替え）をあらかじめ想定しておくこと**。

**再現確認（同日、他5銘柄の調査でも多発）**: `finance.yahoo.co.jp` の個別銘柄ページ・サブページで
9434（ソフトバンク）・4597（ソレイジア・ファーマ）・285A（キオクシアHD）・9984
（ソフトバンクグループ）の調査中にも同様の500エラーが発生（9984では同一URLへの再試行で
200が返るなど間欠的な挙動も観測）。8社中5社で発生しており、単発の偶発障害ではなく
**当日を通じた広範なサービス不調だった可能性が高い**。トップページ・ランキングページは
本日中もWebFetchで取得できていたため、全面障害ではなく個別銘柄ページ系統に偏っている
可能性がある。恒常的な障害かどうかは次回セッションで再確認すること。日経・株探・みんかぶへの
代替で全銘柄とも調査を完了できており、**Yahoo!ファイナンス単独に依存しないマルチソース運用が
有効に機能した**。

### 決算短信PDFのテキスト抽出手法（2026-07-09、企業調査部RSH-S02で確立）

`WebFetch` はPDFバイナリ（決算短信・有価証券報告書等）を直接解析できない。4564
（オンコセラピー・サイエンス）と9434（ソフトバンク）の調査では、以下の手順で一次情報を
直接抽出できた:

1. `curl` で決算短信PDFのURLを取得（会社IRサイトから直接リンクされている場合が多い）
2. `poppler-utils`（`pdftotext`コマンド）で テキストに変換（未インストールの場合は
   `poppler-utils`・`poppler-data` を追加インストールする必要があった）
3. 抽出したテキストから財務数値を読み取り、必要に応じてPythonで再計算（自己資本比率等。
   `corp/CLAUDE.md` 第2章-5「数値の再計算は必ずPythonで行う」に準拠）

一次情報（決算短信・有価証券報告書）に基づく数値は、Yahoo!ファイナンス等の二次情報より
信頼性が高いため、今後の企業調査でも標準手順として採用を検討する価値がある。
