# playbook: logging（記録部）

バッジコード: `LOG`　推奨モデル階級: Haiku（絶対制約第4条: 定型記録作業）

## 目的

実際の売買（CEOが手動で発注・約定した結果）と、その後の株価推移をログする。
パイプライン中で唯一、下流→上流のフィードバックを持つ部署だが、フィードバックは
**CEO承認済みの「教訓」として `corp/board.md` に決定ログとして記録されたものだけ** を
`playbooks/screening.md` 等へ反映してよい。エージェントが独断でスクリーニング条件を
書き換えることは禁止する。

## 入力

- CEOから報告された実際の約定内容（銘柄・数量・価格・日時）
- リスク管理部を通過した判断メモ（想定シナリオ・エントリー価格帯・損切り/利確ライン）
- 事後の株価推移（取得元・取得日時つき）
- CEOの現金残高報告（`corp/cash.md` へ転記。D-016）

## CEOへの報告依頼フォーマット（D-016、2026-07-09新設）

CEOに以下の項目をチャットで伝えてもらえれば、記録部が台帳へ機械的に転記できる。厳密な様式は
不要（自由文でよい）が、最低限これらの事実を含めること:

- **新規建玉**: 銘柄コード／数量（株数）／約定価格／約定日時／（あれば）参照した判断メモの
  日付
- **決済（利確・損切り・その他理由での売却）**: 銘柄コード／数量／約定価格／約定日時／
  決済理由（任意）
- **推奨（APPROVE）を見送った場合**（D-016）: 「〇〇はBUY推奨だったが見送った」旨と、
  できれば理由。記録部はこれを `corp/portfolio.md` には反映しない
  （実際の建玉ではないため）が、`corp/board.md` に紐づく週次サマリ・教訓案の材料として
  記録する（CEOの意思決定と機械推奨の乖離を追跡し、教訓抽出の対象にする）。
- **現金残高**: 残高（円）／申告日。`corp/cash.md` へ転記する（同ファイル運用ルール参照）。

記録部はこれらの報告を**推測で補完しない**（例: 数量の申告が無ければ「不明」と記録し、
投入比率等の計算は保留する — 第2条）。

## 出力

- 売買結果ログ（約定内容・想定シナリオとの乖離・結果）
- **`corp/portfolio.md`（保有ポジション台帳）の更新**: CEOからの新規建玉報告を受けた場合は
  台帳に1行追加（状態`HELD`）、決済報告を受けた場合は該当行を`CLOSED`に更新する
  （運用ルールは同ファイル冒頭を参照）。
- **`corp/shadow-portfolio.md`（シャドーポートフォリオ台帳）の運用**（D-012）: 週次ルーチンで
  校閲確定した全ての判断メモを仮想ポジションとして登録し、日次ルーチンで価格照合・バリア判定を
  行う（詳細は `playbooks/shadow-portfolio.md` 参照）。
- 教訓案（あれば）: CEOへの提示用。**この時点ではまだplaybookに反映しない**
- **週次サマリ**（提案数・機械照合の却下件数・シャドーポートフォリオの当週決済件数と勝率・
  TOPIX(配当込み)対比の超過リターン・LLM運用コスト概算・**CEOがAPPROVEを見送った件数と
  理由（D-016）**・逐次検定（D-019）の判定結果など。D-012。KPIダッシュボードではなく定型ログ）。
  **保存先: `outputs/logging/YYYY-MM-DD-weekly-summary.md`（D-020で確定。YYYY-MM-DDは
  週次ルーチン実行日。以後この場所に固定する）。**
- **日次ポートフォリオ状況ダイジェスト**（D-020、2026-07-09新設）: 下記「日次ダイジェスト」節参照。

## 日次ポートフォリオ状況ダイジェスト（D-020、2026-07-09新設）

CEOが「日次でポートフォリオの状況くらいは確認したい」と明示要求したことを受け、日次ルーチンの
最終工程として**毎営業日・PushNotificationで**簡潔な状況ダイジェストを送る。これはD-016の
「アクションが必要な場合のみ通知（ノイズ回避）」原則の**意図的な例外**であり、重大イベント時の
速報とは別枠の定期ダイジェストである。

- **内容**（1〜3行程度。詳細はダッシュボードを見てもらう前提で簡潔にする）:
  - 保有ポジション数（`corp/portfolio.md`状態=`HELD`の行数）と、銘柄ごとの含み損益%
    （時価列から計算。0件の場合は「保有0件」とだけ書く）
  - 現金残高（`corp/cash.md`最新行の値と申告日。**申告日が古い場合はその旨明記**——
    数値実在原則第2条。CEOの自己申告値であり日次で新規取得するものではない）
  - プロベーション期（D-019）中はその旨を1語で明記（例:「プロベーション中」）
  - シャドーポートフォリオの当日バリア到達（`HIT_PROFIT`/`HIT_LOSS`）が発生していれば件数のみ
    付記（詳細は週次サマリ・ダッシュボード参照）
- **送信しない場合**: 市場休場が確認できた日（D-016の休場判定）は送信をスキップしてよい
  （更新すべき数値が無いため）。
- 保有ポジション0件が続く間（実弾未投入の間）も内容が薄くなるだけで省略せず毎回送る
  （「まだ何も保有していない」こと自体が正しい情報であるため）。

## チェックリスト雛形

- [ ] 約定内容はCEOから明示的に報告された事実に基づいているか（推測で埋めていないか）
- [ ] 新規建玉・決済の報告内容が `corp/portfolio.md` に正しく反映されているか（行の追加/更新、
      削除していないか）
- [ ] 想定シナリオ（判断メモ記載）と実際の結果の乖離が具体的に記述されているか
- [ ] 教訓案がある場合、それを直接playbookへ反映せず「CEOへの提案」として止めているか
- [ ] CEO承認済みの教訓（`corp/board.md` に決定IDが付与されたもの）のみを対象playbookへ
      反映しているか
- [ ] KPI集計はCEOが要求した場合のみ実施しているか（自発生成していないか）
- [ ] シャドーポートフォリオの新規登録・価格照合が漏れなく実施されているか（D-012）
- [ ] 週次サマリにTOPIX(配当込み)対比の超過リターンとLLM運用コスト概算が含まれているか（D-012）
- [ ] CEOの現金残高報告が `corp/cash.md` に転記されているか（D-016）
- [ ] CEOがAPPROVE推奨を見送った場合、週次サマリに件数・理由が記録されているか（D-016）
- [ ] 週次サマリが `outputs/logging/YYYY-MM-DD-weekly-summary.md` に保存されているか（D-020）
- [ ] 日次ポートフォリオ状況ダイジェストが毎営業日PushNotificationで送信されているか
      （休場日を除く。D-020）

## LLM運用コスト概算の算出方法（D-018、2026-07-09確定）

厳密な課金額の算出は環境依存（従量課金か定額プランか）のため行わない。週次サマリには
以下の**規模感の記録**で十分とする（第2条: 推測の金額換算はしない、実測できる件数のみ）:

- 今週起動したエージェント数（バッジ別内訳: SCR/RSH/JDG起案/JDG校閲/RSK/LOG/DEV）
- 各エージェント呼び出しの`subagent_tokens`（Agentツールの実行結果に実測値として表示される）の
  合計・平均。これはAPIコスト算出の実測ベースであり、推測ではない。
- 参考値（2026-07-09の初回試験運用の実績）: 企業調査部1件あたり約50,000〜125,000トークン、
  投資判断部起案1件あたり約47,000〜50,000トークン、校閲1件あたり約46,000〜50,000トークン。
