# playbook: dev（開発部）

バッジコード: `DEV`（D-014、2026-07-09新設）

モデル割当（絶対制約第4条）:
- **設計・実装の変更**（テンプレート・ビルドスクリプト・スキーマの変更）: Sonnet標準。
  アーキテクチャ・UX・チャート設計等の複雑な設計判断は上位モデルを充てる（初版設計は
  Fable5が担当、D-014）。
- **ダッシュボードの再生成**（ビルドスクリプト実行＝決定論的な定型作業）: Haiku可。

## 1. 目的と位置づけ

社内向けの可視化ツール（保有ポジション・シャドーポートフォリオの価格チャート・損益
ダッシュボード）を開発・保守する。

- **憲法第4章のパイプライン（一方向DAG）に含まれない支援部署**である。投資判断
  （BUY/WATCH/PASS）・リスク照合・発注・台帳の数値更新のいずれにも一切関与しない。
  開発部の成果物が上流工程（スクリーニング〜リスク管理）の入力になることはない。
- 開発部の扱うものは**表示だけ**である。台帳・データファイルの数値を生成・更新するのは
  各担当部署（記録部・日次ルーチン実行者）であり、開発部はそれを読み取って可視化する。

### 憲法第2章-7・第8章（自発生成禁止）との関係

本ダッシュボードは**CEOがD-014で明示的に要求した成果物**であり、日次/週次ルーチンでの
再生成はその定常指示の履行である（「自発生成」に当たらない）。ただし、**表示内容を勝手に
拡張しない**: 勝率検定・超過リターン等の統計的KPI集計を新セクションとして追加するのは、
CEOが要求した時だけとする（既存の表示は台帳の生データと固定式による導出値に限る）。

## 2. 厳守事項（絶対制約との関係）

- **第1条（発注禁止）**: 証券会社API・自動売買ツールへの接続コードは一切書かない。
  ダッシュボードは閲覧専用（read-only）であり、発注・注文入力・外部送信につながる
  インタラクティブ機能を実装しない。
- **第2条（数値実在原則）**:
  - 表示する数値の情報源は `corp/data/*.json`（＝台帳・判断メモに記録された取得元・
    取得日時つきの実数値）のみ。ビルド時にJSONの内容を**無加工で**埋め込む。
  - 損益率・バリア距離等の導出値は、埋め込み値からの固定式（閲覧時にJSで計算）に限る。
    新たな数値の推測・補間・捏造は禁止。
  - データが無い状態は「データなし」「算定不能」と明示する（0やダミー値で埋めない）。
    例: 保有ポジション0件の空状態表示、決済0件時の勝率「—」表示。
  - 各データポイントの取得元・取得日時をUI上で確認できるようにする（ツールチップ・
    テーブル列）。
- **第5条（助言限界原則・非公開）**: ダッシュボードはCEO専用の社内ツールであり、第三者への
  公開・配布を前提としない。投資助言的な表現（「買い時」「割安で妙味」等）を画面上に一切
  含めない。事実（価格・損益・判定区分・バリアとの距離）の可視化に限定し、免責文言
  （第5条の文言）をフッターに常時表示する。

## 3. 入力

- `corp/data/price-history.json`（保有ポジションの価格時系列）
- `corp/data/shadow-price-history.json`（シャドーポートフォリオの価格時系列）
- （参照のみ）`corp/portfolio.md`・`corp/shadow-portfolio.md`・`corp/watchlist.md` —
  データファイルと台帳の整合確認用。食い違いがあれば**台帳を正**とし、データファイルの
  修正は台帳を更新した部署（記録部・日次ルーチン実行者）へ差し戻す（開発部は数値を直さない）。

### データスキーマ（両JSONで共通）

```
{
  "_comment": "<説明>",
  "as_of": "YYYY-MM-DD",                 // データ基準日
  "positions": {
    "<ポジションID>": {                   // 保有: 台帳行に対応するID / シャドー: SP-XXX
      "code": "銘柄コード",
      "name": "銘柄名",
      "verdict": "BUY|WATCH|PASS",       // シャドーのみ
      "state": "OPEN|HIT_PROFIT|HIT_LOSS|EXPIRED",  // 保有はHELD|CLOSEDを使用
      "entry_price": <数値>,              // 保有: 約定価格 / シャドー: 判定基準日株価
      "entry_date": "YYYY-MM-DD",        // 省略時はpoints[0].dateを表示に使う
      "stop_loss": <数値>,                // -10%バリア（記録時点の値で固定）
      "take_profit": <数値>,              // +25%バリア（同上）
      "confidence": "高|中|低",           // 保有のみ（D-011）
      "weight_pct": <数値>,               // 保有のみ: 投入比率（%。corp/portfolio.md の
                                         //   台帳転記値。D-015のPML・ストレス表示に使用）
      "settle_date": "YYYY-MM-DD",       // 決済済みのみ
      "settle_price": <数値>,             // 決済済みのみ
      "note": "構造的注記（D-013等）",     // 任意
      "points": [                        // 価格時系列（日次ルーチンで追記、末尾が最新）
        { "date": "YYYY-MM-DD", "price": <数値>,
          "source": "取得元", "source_time": "取得日時" }   // 第2条: source系は必須
      ]
    }
  }
}
```

- 追記は**日次ルーチンの価格照合工程**（`corp/RUNBOOK.md` 日次3・5）が行う。開発部は
  スキーマの番人であり、値の書き手ではない。
- 価格履歴は当面営業日1点・無期限保持。間引き方針は未確定（§7）。

## 4. 出力（成果物）

- **正本**: `corp/dev/dashboard.html` — 自己完結型HTML（外部リクエストなし。CDN・Webフォント・
  外部画像・fetch/XHRを一切含まない。CSS/JS埋め込み、データはビルド時に埋め込み）。
  ブラウザでファイルを直接開いて閲覧できる。
- **ソース**（開発部の管理対象）:
  - `corp/dev/dashboard.template.html` — UX・チャート・スタイルの定義。プレースホルダ
    `var DATA = __EMBED_DATA_JSON__;` を含む。
  - `corp/dev/build_dashboard.py` — 両JSONをテンプレートに埋め込んで正本（完全なHTML文書、
    `<html>/<head>/<body>`込み）を生成する決定論的スクリプト。
  - `corp/dev/build_artifact_fragment.py` — 正本から `<title>/<style>/<body>` の中身のみを
    抽出し、Artifact公開用の断片 `corp/dev/dashboard-artifact-fragment.html` を生成する
    決定論的スクリプト。**Artifact機能は独自に`<html>/<head>/<body>`を付与するため、
    正本（完全なHTML文書）をそのまま渡すとタグが二重になる。Artifactへ公開する際は
    必ずこのスクリプトの出力を使うこと。**
- **常設閲覧URL（Artifact）**: 2026-07-09にCEOの直接指示（本playbookの新設依頼と一体）を
  受けて初回デプロイ済み。URL: `https://claude.ai/code/artifact/e1a2da41-fbdc-42cb-839f-cb79271caed5`
  （非公開。共有はCEOがページの共有メニューから行う）。
  **再デプロイ時は必ず `file_path` に `/home/user/corp/corp/dev/dashboard-artifact-fragment.html`
  （リポジトリ管理下の正本パス）を指定すること。** 初回デプロイ時に一時スクラッチパスから
  公開したため、同一のリポジトリパスから再公開した際にURLが変わった経緯がある
  （旧URL `.../545e813a-...` は失効・参照しないこと）。以後はこのリポジトリパスを
  一貫して使う限り同一URLが維持される。
- 変更はCEO承認不要（自動push可、憲法第8章）。データ・実装いずれかが更新されたら
  `build_dashboard.py` → `build_artifact_fragment.py` の順に実行し、出力ファイルを
  **同一URL（file_pathを変えない）** で再デプロイする（第5条: 非公開設定のまま）。

## 5. ダッシュボードの仕様（現行設計の要点）

1. **保有ポジション（実運用）**: 状態=HELDをカード表示。**0件の場合は空状態カード**
   （「現在保有ポジションはありません」＋データが載る条件の説明）を表示し、推測値では
   埋めない。
   - **リスク枠タイル（D-015）**: 保有セクション冒頭に常時表示する4タイル。
     ①サイジング方式（「比例（現在資金残高基準）」＝D-015のルール記述の転記）、
     ②PML上限5.0%（ルール定数、出典D-015）、③現在のPML＝状態HELDのΣ(weight_pct×10%)、
     ④ストレスシナリオ＝Σ(weight_pct×25%)（全HELD同時-25%ギャップの**仮定**であることを
     タイル内に明記）。③④は台帳転記値`weight_pct`からの固定式導出（第2条）。HELD 0件の
     場合は「0.00%（保有0件）」、`weight_pct`欠落があるHELD行が1件でもあれば「算定不能」
     と表示する（推測で埋めない）。
2. **シャドーポートフォリオ**: KPIタイル（追跡中OPEN件数／判定内訳BUY・WATCH・PASS／
   決済済み件数／勝率＝決済0件なら「—」＋約102件目安の注記）＋全件のカードグリッド。
   各カード: 判定区分バッジ・現在値・対エントリー損益率・価格推移チャート（銘柄ごとの
   small multiples。価格レンジが銘柄間で3桁以上違うため共通軸には載せない）・
   損切り/利確バリアとの距離（現在値比%）・D-013等の構造的注記。
3. **チャート**: 単一系列の折れ線（カテゴリカルslot1固定）。エントリー基準線（実線・
   ニュートラル）と損切り（critical・破線）/利確（good・破線）の閾値線をy域に含める。
   ホバーで最近傍ポイントにクロスヘア＋ツールチップ（日付・価格・**取得元・取得日時**）、
   各点はキーボードフォーカス可能。損益方向の色（good/critical）は損益・バリアにのみ
   使い、系列色には使わない。
4. **アクセシビリティ**: 凡例常設、全件テーブルビュー（チャートの代替）、テキストは
   インクトークンのみ（系列色の文字なし）、`textContent`によるDOM構築（innerHTML連結禁止）。
5. **テーマ**: `prefers-color-scheme` と `data-theme` 属性（`dark`/`light`）の両対応。
   両モードとも dataviz 参照パレット（検証済み）を使用。
6. **フッター**: 第5条の免責文言・非公開注記・数値出典の説明・シャドーの解釈上の注意
   （WATCH/PASS判定の正誤を直接示すものではない）を常時表示。

## 6. 手順

### 6-1. 再生成（定型・Haiku可。日次/週次ルーチンから呼ばれる）

1. 日次/週次ルーチンで `corp/data/*.json` が更新されたことを確認する（更新が無ければ
   再生成不要）。
2. `python3 corp/dev/build_dashboard.py` を実行する（出力メッセージで件数・as_ofを確認）。
3. `python3 corp/dev/build_artifact_fragment.py` を実行し、Artifact公開用断片を再生成する。
4. §6-3のチェックリストの「再生成時」項目を確認する。
5. `corp/dev/dashboard-artifact-fragment.html` を、§4記載の同一URL（file_pathを変えない）へ
   再デプロイする。

### 6-2. 設計・実装の変更（非定型・Sonnet以上）

1. 変更はテンプレート／ビルドスクリプトに対して行う（`dashboard.html` を直接編集しない —
   次回ビルドで消える）。
2. チャート・配色は dataviz スキルの手法に従う: フォームは目的から選ぶ／色は最後・
   役割で決める／カテゴリカル色は固定順／ステータス色は良し悪しの意味がある場合のみ／
   ホバー・凡例・テーブル代替を用意する。**パレットを変更した場合はパレット検証
   スクリプト（validate_palette.js）をlight/dark両モードで実際に実行**し、結果を
   変更コミットに記録する。
3. 変更後は必ずビルドを実行し、実際にレンダリングして両テーマを目視確認する
   （ヘッドレスブラウザのスクリーンショットでよい）。

### 6-3. チェックリスト

再生成時（毎回）:
- [ ] 表示数値の出典が `corp/data/*.json`（＝台帳の実数値）に限定されているか（第2条）
- [ ] 空状態（保有0件・決済0件等）が「データなし／算定不能」表示になっているか
      （ダミー値で埋めていないか）
- [ ] as_of・件数がビルド出力メッセージと台帳の件数に一致しているか
- [ ] 再デプロイする場合、同一URLか・非公開設定のままか（第5条）

実装変更時（追加で）:
- [ ] 発注・自動売買・外部送信につながる機能を実装していないか（第1条）
- [ ] 外部リクエスト（CDN・Webフォント・fetch等）を含まない自己完結型HTMLか
- [ ] 免責文言（第5条）がフッターに表示されているか
- [ ] 投資助言的な表現が画面上に無いか（事実の可視化に限定されているか）
- [ ] パレット変更時、検証スクリプトをlight/dark両モードで実行しPASSしたか
- [ ] 両テーマ（ライト/ダーク、`data-theme`上書き含む）をレンダリングして目視確認したか
- [ ] ツールチップ・テーブルで取得元・取得日時が確認できるか（第2条）
